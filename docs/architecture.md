# 2Dピクセルアート UV写像マルチレイヤーアニメーションシステム 詳細仕様書

## 1. システムアーキテクチャ概要
本システムは、2Dゲーム開発におけるピクセルアートのアニメーション（動き）とテクスチャ（外見）を完全に分離する「UV写像ルックアップテクスチャ（LUT）技術」を利用したグラフィックパイプラインです。

DCCツール（Aseprite）側で「アニメーションの動き（UV座標）」を自作拡張機能によりベイクし、ゲームエンジン（Unity URP 2D）のフラグメントシェーダー内で「ベーススキン」「影」「オプション状態」の複数のLUTレイヤーをリアルタイムに合成・描画します。

対象環境: Unity `6000.5.4f1`（Unity 6）、URP `com.unity.render-pipelines.universal` `17.6.0`。対象プラットフォームはPCのみ。想定される同時描画キャラクター数は最大5〜6体程度で、大規模な同時描画（群衆等）は想定しない。

## 2. アセットの定義とデータ構造
システムは、以下の6つの画像アセット／概念で構成されます。

### 2.1 パーツ判定用カラーパレット（固定・全キャラ共通）
`Overlay`がどのパーツを表しているかを判定するための、**6色固定のグローバルパレット**です。プロジェクト全体で1つだけ定義し、拡張機能内にハードコード（またはドキュメント化）します。キャラクターごとに新規アセットを用意する必要はありません。

パーツ構成は現状固定です（将来的にキャラクターごとの可変対応も検討するが、現時点では以下で確定）:

```
[頭, 体, 右腕, 左腕, 右足, 左足]
```

### 2.2 Map（マップ / 展開図）
キャラクターの各パーツを展開し、実際の色データ（`LUT`）の参照位置をピクセル単位の色で表現するマスター参照画像です。

- 解像度は最大 **256×256px** に制限されます（§4.1のUV座標エンコード方式に起因）。
- 色の一意性は**パーツ内でのみ**保証されていれば十分です（パーツをまたいだ完全な一意性は不要）。これは、パーツの判定を先にOverlay（§2.4）が担うため、Map側の色照合はパーツ内スコープに限定して行われることによります。
- 配色は**意図的な空間グラデーション**にします（例: パーツの付け根から先端に向かって色相・明度を系統的に変化させる）。この向きの規約は全パーツ・全キャラクター共通で固定します。これは、アーティストが`DirectionMap`（§2.5）を手描きする際、目視で「だいたいこのあたりの色」を推測しやすくするためのものです。
- 背景・未使用領域は透明（アルファ0）で表現し、「どのパーツにも属さない」ことを示します。

### 2.3 LUT（ルックアップテクスチャ / スキンマップ）
Mapと全く同じ解像度・レイアウトで作成された、実際のカラーを持つ静止画テクスチャです。標準的なカラーテクスチャとして扱うため、**sRGBは有効**（Linear無効）でインポートします。

1キャラクターにつき、**固定3枚1組**のLUTレイヤーを用意します。レイヤー数は可変にせず、全キャラクター共通でこの3枚構成です。

- `BaseSkin_LUT`: 基本の肌や服の色
- `Shadow_LUT`: 落ち影などの乗算合成用レイヤー
- `OptionalState_LUT`: 用途を限定しない汎用レイヤー（ダメージ表現、状態異常表現など、ゲームロジック側で自由に転用する）

1キャラクターは将来的に複数のスキンバリエーション（衣装・色違いなど）を持てるようにします。スキンバリエーションごとに、このLUT3枚1組を追加で持たせられるデータ構造にします。

LUTとキャラクター／スキンの紐付けは、命名規則による自動解決ではなく、**キャラクターごとに手作業で用意するScriptableObject**にテクスチャ参照を直接ドラッグ＆ドロップして設定します。またLUTインポート時には、対応するMapとの解像度不一致をEditor拡張／AssetPostprocessorで自動検証します。

### 2.4 Overlay（オーバーレイ / パーツ判定アニメーション）
アーティストが手作業で描画する、アニメーションごとのスプライトシートです。

- 色指定の制約: §2.1の**固定6色パレットのみ**を使用し、パーツごとに1色でベタ塗りします。パーツ内の位置情報は一切表現しません。
- 役割は「このピクセルはどのパーツか」の判定のみであり、グラデーションや指向性のある塗り方は不要です。見た目の質を気にせず、素早く塗ることができます。

### 2.5 DirectionMap（指向性マップ / 位置情報アニメーション）
Overlayと**同一の`.aseprite`ファイル内の別レイヤー**として、同時に手描きするアニメーションスプライトシートです（同一キャンバス・同一フレームで描くことで、Overlayとのシルエット・タイミングのズレを防ぎます）。

- 色指定の制約: 実際のキャラクターの色ではなく、必ず`Map`で定義されたインデックスカラーのみを使用して描画します。
- 指向性ペイントの適用: 腕の回転やパーツの伸縮などの向き（指向性）を一意に追跡できるよう、単一のベタ塗りではなく、Mapのグラデーション配色（§2.2）を目視で参照しながらグラデーションやストライプ模様を用いて描画します。
- **これは完全な手作業です**。Asepriteの変形ツール（回転・拡縮等）でMapの基準グラデーションを機械的に転写する案も検討しましたが、関節を曲げるような非アフィン変形で誤差が出るリスクがあるため採用せず、手描きに統一しています。

### 2.6 UV Sprite（ソース / rgアニメーション）
`Overlay`・`DirectionMap`・`Map`を照合し、自作のAseprite拡張機能によって自動生成される最終的なアニメーションデータです。人間の目にはほぼ黒一色の画像として見えます。

- B/Aチャンネルは現時点で未使用・未定義です（将来用途が出た場合に再検討します）。

## 3. ワークフローと責務の分離

### 3.1 オフラインフェーズ（Aseprite・自作拡張機能）
Unityでのランタイム負荷を軽減するため、UV座標のベイクはすべてDCCツール内でオフライン処理として完結させます。ベイク拡張機能は**自作**します（既存の拡張は使用しません）。

1. アーティストは`Map`、`LUT`一式を作成します。`Map`は§2.2の配色規約（パーツ内一意・意図的なグラデーション）に従います。
2. アーティストは1つの`.aseprite`ファイル内に`Overlay`レイヤー（§2.1の6色パレットでパーツをベタ塗り）と`DirectionMap`レイヤー（Mapの色を参照した指向性ペイント）を同時に描画します。拡張機能は、Map参照表示やベイク前プレビューなど、この手作業を支援する機能を提供します。
3. ベイク処理（拡張機能内、下記アルゴリズム）により`UV Sprite`を自動生成し、Unity向けの最終出力スプライトシートとしてエクスポートします。

**ベイクアルゴリズム**（ピクセル単位）:
1. `Overlay`の色を§2.1の6色パレットと照合し、対象パーツ K を判定する。
2. 同じ座標の`DirectionMap`の色を、パーツ K に属する`Map`の色域内でのみ照合し、元の`Map`座標 (X, Y) を特定する。
3. (a)(b)いずれかで一致する色が見つからない場合は**何もしない**（エラー表示なし）。当該ピクセルは`UV Sprite`上で**透明**として出力される。これはAA（アンチエイリアス）による色の滲みも自動的に弾く仕組みとして機能する（AA混色は既存のパーツ色・Map色のどれとも一致しないため）。
4. 一致した (X, Y) を8bitで正規化し、X→Rチャンネル、Y→Gチャンネルに書き込む。座標原点はUnity基準（左下、V=0が下）に統一し、**Y軸反転はこの拡張機能側で行う**（Unity・シェーダー側での補正は不要）。

`Overlay`・`DirectionMap`のアートワークにはアンチエイリアスを使用しません（ハードエッジのみ）。専用の強制チェック機構は設けず、AAが混入した場合は上記3.の「一致しない→透明」という形で自然に検出されます。

アニメーションの尺・フレームレートは特に標準化しません。アニメーションクリップごとに自由に設定できます。

### 3.2 ランタイムフェーズ（Unity）
Unityは、画像の生成やベイクといった事前処理は行わず、シェーダーによる「動的なカラー割り当てと合成」のみを担当します。

1. `UV Sprite`を通常のスプライトとして配置し、`Animator`で再生します。Animatorのstate名・パラメータ名は全キャラクター共通の命名規則に統一し、`SetSkin(index)` / `SetParameter(value)`のような汎用APIで制御できるようにします。
2. キャラクター初期化時、対応する`LUT`3枚をキャラクター専用のマテリアルインスタンス（`Renderer.material`で自動複製）にセットします。オブジェクトプーリングは現時点で採用せず、通常の`Instantiate`/`Destroy`で扱います。
3. フラグメントシェーダー内で、再生中の`UV Sprite`から読み取ったRG値を座標として、各`LUT`をリアルタイムにサンプリングして画面に出力します。

`UV Sprite`・`LUT`ともに**Sprite Atlasは使用しません**。同時描画キャラクター数が5〜6体程度と少なく、キャラクターごとにマテリアルインスタンスが分かれる設計（後述§5）と組み合わせるとAtlas化によるドローコール削減効果が実質的に得られないためです。

半透明・フェードアウトなどキャラクター全体の透過表現、およびURP 2D Light（`Light2D`）との統合は、現時点でスコープ外とします。

LUTデコード結果を確認するためのプレビューツールを、Unity Editor拡張として別途用意します。

## 4. シェーダー実装詳細 (Unity URP 2D HLSL)

### 4.1 座標デコードとテクセルオフセット補正
UV Spriteからサンプリングした正規化カラーデータ（R, G）から、LUT上の正確なUV座標を復元する際、浮動小数点の丸め誤差による「テクセル境界からの色漏れ（グリッチ）」を防ぐための数学的補正が必須です。

シェーダー内でのデコード計算式には、必ず **「ハーフテクセルオフセット（+0.5）」** を加算し、テクセルの中央を正確にサンプリングするように設計します。座標原点はUnity基準（左下）に統一済みのため、シェーダー側でV軸反転などの追加補正は不要です。

```hlsl
// LUTの幅と高さ (CBUFFERから取得、またはTexelSizeを利用)
float lutWidth = _BaseSkinLUT_TexelSize.z;
float lutHeight = _BaseSkinLUT_TexelSize.w;

// RG値から元のインデックス座標を復元し、+0.5のオフセットを加算して正規化
float u = (floor(sourceColor.r * 255.0) + 0.5) / lutWidth;
float v = (floor(sourceColor.g * 255.0) + 0.5) / lutHeight;
float2 lutUV = float2(u, v);
```

### 4.2 マルチレイヤーのリアルタイム合成
フラグメントシェーダー内で、復元した lutUV を用いて固定3枚のLUTから色を取得し、動的にブレンドします。ブレンド方法はレイヤー種別ごとに決め打ちで、データ駆動のブレンドモードシステムは導入しません。

- `BaseSkin_LUT`: ベースカラー
- `Shadow_LUT`: 乗算合成
- `OptionalState_LUT`: **乗算・加算の2経路のみ**選択可能（用途に応じてマテリアルインスタンスごとに設定する、CBUFFER互換のint/enumプロパティで切り替え）。これ以上の汎用化（任意ブレンドモード対応等）は行いません。

```hlsl
// 1. 各LUTからサンプリング
half4 baseColor = SAMPLE_TEXTURE2D(_BaseSkinLUT, sampler_BaseSkinLUT, lutUV);
half4 shadowColor = SAMPLE_TEXTURE2D(_ShadowLUT, sampler_ShadowLUT, lutUV);
half4 optionalColor = SAMPLE_TEXTURE2D(_OptionalStateLUT, sampler_OptionalStateLUT, lutUV);

// 2. 動的なブレンド処理
half4 finalColor = baseColor;
finalColor.rgb *= lerp(1.0, shadowColor.rgb, _ShadowIntensity);

// OptionalStateは乗算/加算の2経路（_OptionalStateBlendMode: 0=乗算, 1=加算）
half3 optionalMultiply = lerp(1.0, optionalColor.rgb, _OptionalStateIntensity);
half3 optionalAdditive = optionalColor.rgb * optionalColor.a * _OptionalStateIntensity;
finalColor.rgb = lerp(finalColor.rgb * optionalMultiply, finalColor.rgb + optionalAdditive, _OptionalStateBlendMode);

// 元のUVスプライトのアルファを適用して出力
return finalColor * sourceColor.a;
```

`_ShadowIntensity` / `_OptionalStateIntensity` / `_OptionalStateBlendMode` などのパラメータをキャラクターごとに異なる値でリアルタイムに変化させたい場合は、§5.2の通りキャラクターごとのマテリアルインスタンスを用いて`SetFloat`/`SetInt`で設定します。

なお、Sprite Atlasは使用しないため（§3.2）、アトラス空間補正（`_UVRemap`）は実装しません。

## 5. パフォーマンスと最適化の制約

### 5.1 SRP Batcher対応の厳守
URP環境においてパフォーマンスを維持するため、`Renderer.SetPropertyBlock()`（`MaterialPropertyBlock`）の使用は禁止とします。これを使用すると、SRP Batcherの対象から外れ、ドローコールが増大してCPUボトルネックを引き起こします。

### 5.2 CBUFFERの設計要件
カスタムシェーダーをSRP Batcherと互換させるため、以下のルールに従って変数を宣言します。

- すべてのカスタムマテリアル変数（`_ShadowIntensity`、`_OptionalStateIntensity`、`_OptionalStateBlendMode` など）は、単一の定数バッファ `CBUFFER_START(UnityPerMaterial)` 内に定義すること。
- テクスチャサンプラー（`Texture2D`）はCBUFFERの外部で宣言すること。
- **スキン（LUT）の切り替え運用**: 個々のキャラクターごとに異なるLUTセットを適用するため、`Renderer.material`によるマテリアルインスタンスの自動複製を利用し、キャラクターごとに個別のマテリアルインスタンスを保持します。`MaterialPropertyBlock`は使用しません。
- Sprite Atlas未使用・キャラクターごとのマテリアルインスタンス採用という構成上、キャラクター間でのドローコール統合（バッチング）は行われません（1キャラクター＝1ドローコールが基本）。同時描画数が5〜6体程度である前提のもとで許容しています。同時描画数が大幅に増える場合は、この設計方針自体を再検討する必要があります。
- キャラクターのGameObjectを破棄する際は、`Renderer.material`で複製されたマテリアルインスタンスが残らないよう明示的に破棄することが望ましい（オブジェクトプーリングは現状未採用）。
